Object Literals

Ruby Hashes vs. JavaScript Objects

March 3rd, 2016

So after 4 weeks of getting close to Ruby, I miss her oh so much. It was a relationship that grew quick with emotional struggle. At first there were times when I thought it wasn't going to work out, but then when I started to really get to know Ruby, she showed me all the wonderful things she could do. As soon as I got comfortable with her and began to think of all the possiblities of our future together, Ruby was taken away from me and JavaScript showed up. I'm still blogging about programming languages, don't worry. Sorry if that made you uncomfortable, but it's true. Ruby has so many built-in abilities, like methods that really make it a beautiful language. Programmers talk about "Syntactic Sugar", which to me is just a romantic way of saying readability, and Ruby has that. JavaScript has many differences from Ruby that make it less readable, and as a result, harder to get a long with.

So begins my new relationship with JavaScript. It was a struggle this week learning another language after getting used to the first, but then I remind myself of how hard it was with Ruby in the beginning. So I stopped looking at what makes JavaScript different in a negative way, and started to focus on how JavaScript is similar. One aspect of Ruby that was hard for me to master was the Hash. An Array was pretty straight forward as a collection of objects, but using a Hash as a collection took me some time. That was why one difference that kind of made me happy to see in JavaScript is that there isn't really a data type just like Ruby's Hash. But you see I worked so hard to understand a Hash and how to use it, iterate through it and manipulate it, that I kind of missed it. So after some further research I found that JavaScript actually has something that can be used in a similar fashion. I give you JavaScript's own object literals.

The definition os an object literal is "a list of zero or more pairs of property names and associated values of an object, enclosed in curly braces ({})." Yup, sounds like a Hash, but let's delve a little deeper to actually visually see the difference in how object literals behave compared to a Hash. First let's create a Hash: 

        
            meals = {breakfast: "eggs", lunch: "sandwich", dinner: "pasta"}

Now let's look at a JavaScript object literal: 

        
            var meals = {breakfast: "eggs", lunch: "sandwich", dinner: "pasta"};

These seem pretty much identical besides some minor syntax differences. We have a Ruby Hash meals that has keys and their corresponding values. What we have in JavaScript is something different. We defined an object meals that has properties. The properties of the object are variables themselves, making them a little more limited as data types. Another interesting difference is how we access these properties in JavaScript vs. how we access the values of a Hash key in Ruby. 

        
            # Using Ruby
            meals[:lunch]
            => "sandwich"

            // JavaScript
            meals.lunch;
            => sandwich

Believe it or not, we actually could have used bracket notation for the JavaScript object as well. In our example abouve we could have called the lunch property by typing meals["lunch"], but that is only because we called it as a string, meals[lunch] in JavaScript would return an error. Dot notation is more common and easily readable, but bracket notation allows us to access properties containing special characters such as [].

There's a lot more similarities between Ruby Hashes and JavaScript objects and between the laguages in general that really prove to me that one language is not necessarily better than the other. At this point I still prefer Ruby, but I believe that is solely based on familiarity. Once I take what I have learned from Ruby and find similarities in JavaScript, and see all the fun things it can do with web developement, I'm sure I will begin to find aspects of JS that I love just as much. Oops, I just called it JS, I think I'm bonding to it already.
