Debugging with Test Codes

An introduction to Test-Driven Development

March 12th, 2016

"If debugging is the process of removing bugs, then programming must be the process of putting them in." - Edsger W. Dijkstra

Some of you may be familiar with this quote. As you begin to delve deeper into the world of programming, you are bound to run into it. It's funny and pretty close to the truth. There is however, one form of coding dedicated to catching bugs right when they are put in before they are committed and this is Test-Driven Development. The Wikipedia definition for Test-driven development(TDD) is "a software development process that relies on the repetition of a very short development cycle: first the developer writes an (initially failing) automated test case that defines a desired improvement or new function, then produces the minimum amount of code to pass that test, and finally refactors the new code to acceptable standards." So basically we develope a test for our code initially before the actual code is developed in order to build an expectation of what we want our code to meet. Once the expectation or test code is built we can move onto our actual program and test as we code. Usually the test code initially fails because "Testing can only prove the presence of bugs, not their absence."Once the code passes the test the test code gets out of the way and lets the actual program do what is expected.

The basic TDD cycle is:

1. Add a test
2. Run all tests and see if the new one fails
3. Write some code
4. Run tests
5. Refactor code to fit tests
6. Repeat process for each new test

The most popular testing framework in Ruby is called RSpec, and for JavaScript, it is called Jasmine. It is also an expectation in the real world that you will be able to create your own test code with things like assert statements for Ruby, but this doesn't really fall into TDD because you are creating tests after your actual program code has been created.
